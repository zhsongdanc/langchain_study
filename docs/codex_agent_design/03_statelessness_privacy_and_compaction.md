# 03. 无状态请求、隐私边界与上下文压缩，其实是一组连在一起的设计

## 表面看是三个话题，本质上是同一个问题

这篇文章里有三个点容易被分开看：

- 不使用 `previous_response_id`
- 兼容 ZDR
- 需要对超长上下文做 compaction

但如果从架构上看，它们其实都在回答同一个问题：

当 agent 会话不断变长时，状态应该存在哪里，由谁负责，以及怎样在不失控的前提下继续运行。

## 一：为什么宁可请求变大，也要保持无状态

文章提到 Codex 当前没有依赖 `previous_response_id`，而是每次请求自己发送完整上下文。

这看起来有点“反效率”，但背后有很强的工程逻辑。

### 1. provider 依赖更少

如果你的 agent 运行依赖服务端持有会话状态，那你会遇到这些问题：

- provider 必须可靠存储并索引你的历史会话
- 客户端和服务端需要对会话生命周期达成一致
- 迁移到别的 provider 时，状态协议可能不兼容

改成无状态请求后，客户端自己持有状态，服务端只负责本轮推理，边界会清楚很多。

### 2. 隐私边界更清晰

对企业用户来说，最敏感的问题之一不是“模型强不强”，而是“谁保留了我的上下文”。

无状态请求的优势是：

- 责任边界清楚
- 数据驻留策略更容易解释
- 更容易适配零数据保留场景

这点很重要，因为企业系统最终要过的往往不是 demo，而是安全、合规和采购。

### 3. 可移植性更强

只要 provider 支持 Responses 风格接口，runtime 就更容易迁移过去。状态不锁在某个服务端，会让整套 agent 框架更像一个独立 runtime，而不是某个单一厂商的前端壳。

## 二：为什么 compaction 不是“做个摘要”那么简单

长会话 agent 一定会碰到上下文窗口问题。简单做法通常是：

- 丢弃旧消息
- 或者生成一段摘要再继续

Codex 更成熟的地方在于，它把 compaction 当成正式的状态迁移机制。

这意味着 compaction 需要满足的不只是“短一些”，还包括：

- 保留足够的任务连续性
- 尽量保留工具调用后的关键事实
- 让后续轮次还能顺畅接续
- 在必要时保留更深层的模型可用信息

所以这不是“压缩文本”，而是在做“可继续执行的工作状态再表达”。

## 三：ZDR 和 `encrypted_content` 暗示了一个很有意思的双层状态设计

文章里有个容易被忽略但很关键的点：即使在 ZDR 场景下，之前轮次的一些专有推理内容仍能通过 `encrypted_content` 被后续使用。

这说明系统可能在维护两层状态：

- 面向用户和应用层可见的状态
- 面向模型续跑能力保留的状态

这里的精彩之处在于，它没有简单地在“完全透明”和“完全黑盒”之间二选一，而是做了分层：

- 对用户展示 reasoning summary，提升可解释性
- 对系统保留更强的续跑能力
- 同时尽量不破坏隐私承诺

这是一种非常现实的系统折中。

## 这一整组设计为什么值得学

因为它体现了一种成熟 agent runtime 必须面对的现实：

### 1. 状态一定会膨胀

只要 agent 真做事，状态就会越来越多。你不能假设会话永远很短。

### 2. 状态不能随便托管给别人

一旦你面对多 provider、企业部署、隐私要求，状态归属就会成为架构问题，而不是接口细节。

### 3. 状态也不能无限保留原样

上下文窗口有限，所以必须引入状态迁移机制，也就是 compaction。

于是，无状态请求、隐私保护和上下文压缩就连成了一条完整链路。

## 如果你自己做 agent，最应该带走什么

- 从一开始就明确“状态归谁管”，不要后面再补。
- 不要把 compaction 看成摘要插件，把它看成状态迁移模块。
- 对外可见状态和对模型有效状态，必要时可以分层设计。
- 如果面向企业场景，隐私边界最好在 runtime 级别体现，而不是靠文档解释。

## 一句话总结

Codex 这里最强的地方是：

它把“长会话状态管理”当成一整个系统问题来处理，而不是零散地补几个优化开关。
