# 04. 从 Codex Runtime 可以学到的更大架构原则

## 先说结论

如果把文章里所有细节收束起来，可以看到 Codex 的设计在反复贯彻几个大原则：

- runtime 比单次 prompt 更重要
- 状态组织比一次模型调用更重要
- 系统边界比功能堆叠更重要
- 可迁移性和可运维性不是附属需求，而是主设计约束

这些原则非常值得自己做 agent 平台时参考。

## 一：agent 的核心资产不是模型，而是 runtime

模型当然重要，但模型会变，接口会变，provider 也会变。

更稳定、更有复利价值的往往是 runtime 层能力，比如：

- 如何组织会话状态
- 如何处理工具调用
- 如何管理上下文窗口
- 如何压缩长链路任务
- 如何做权限、环境和执行控制

Codex 能同时跑在 OpenAI 托管端点、本地兼容端点、Azure 等环境上，本质上说明他们把“agent 框架”从“某个模型的薄封装”里抽离出来了。

这是非常重要的架构判断。

## 二：环境本身也是 prompt 的一部分

文章里有个隐含但非常关键的思想：工作目录、权限模式、sandbox 配置这些传统上属于“运行环境”的东西，也被显式编码进消息流里。

这很有价值，因为模型的行为往往不只由用户任务决定，也由执行环境决定。

一旦你把环境变化显式写进状态：

- 模型对当前约束的理解更稳定
- 审计时能知道它是在什么权限下行动的
- 配置变化可以进入同一个状态机

这比“框架心里知道，但模型上下文里没体现”更稳健。

## 三：可观察性不是锦上添花，而是 agent 的基础设施

当系统会自动调工具、修改文件、执行多轮推理时，可观察性就不再是可选项。

Codex 的状态建模天然支持：

- 看见每轮工具调用
- 看见配置切换
- 看见压缩前后状态如何变化
- 看见最终输出是如何逐步形成的

这意味着调试 agent 不再只能靠猜 prompt，而能靠完整执行轨迹分析。

对于生产级 agent，这是决定可维护性的关键差别。

## 四：很多“看起来不高级”的地方，其实最难复制

比如：

- 稳定排序工具定义
- 避免中途破坏缓存前缀
- 处理配置更新时尽量 append
- 在隐私约束下维持续跑能力

这些都不炫，但它们恰恰是最接近生产现实的部分。

真正难的通常不是让 agent “看起来能工作”，而是让它：

- 连续工作很久
- 成本不失控
- 行为可解释
- 能跨环境部署
- 出问题时可定位

这也是为什么很多 demo agent 和成熟 agent 之间差距巨大。

## 五：如果你要自己实现简化版 Codex，优先级应该怎么排

### 第一层：先把 runtime 骨架搭对

- 结构化状态表示
- append-only 消息流
- 工具调用和工具输出入日志
- 环境上下文显式化

### 第二层：再处理长会话问题

- prompt 前缀稳定
- 工具顺序稳定
- 配置变化追加表达
- 压缩机制

### 第三层：最后再做 provider 抽象和隐私边界

- inference adapter
- 无状态请求能力
- 企业部署友好的状态策略

这三个层次的顺序很重要。很多人会先想“接哪个模型”，但更值得先想的是“状态如何流动”。

## 一句话总结

Codex 这篇文章最精辟的启发是：

做 agent 的关键不是把 LLM 接上工具，而是围绕状态、缓存、压缩、隐私和环境约束，做出一套长期可运行的 runtime。
