# 02. 为了命中 Prompt Cache，主动把 Runtime 设计成“前缀稳定”

## 这是文章里最值得借鉴的点之一

很多人一看到“每轮都把历史上下文重新发给模型”，第一反应是：

- 这不是很浪费吗？
- 请求体不是会越来越大吗？
- 为什么不直接依赖服务端会话状态？

这些问题都合理，但 Codex 的答案非常工程化：真正昂贵的不是网络传输，而是模型采样。于是它优先优化的是 prompt cache 命中率，而不是表面上的请求体大小。

## 核心思想：让旧 prompt 尽量成为新 prompt 的精确前缀

Prompt cache 的命中依赖一个很苛刻的条件：新请求的前缀必须和旧请求精确一致。

这意味着只要你做了下面这些事，缓存就可能失效：

- 改了工具列表顺序
- 改了系统指令位置
- 动态插入了一些本可以后置的信息
- 中途切换模型
- 让环境描述在 prompt 中间发生漂移

所以，Codex 不是简单地“有缓存”，而是在 runtime 设计上主动迎合缓存机制。

## 这反映了什么系统思维

### 1. 优化目标选得很准

很多系统会本能地先优化容易看到的指标，比如：

- HTTP payload 大小
- 请求次数
- 序列化开销

但在 LLM agent 里，真正的大头常常是模型推理成本。Codex 的设计说明他们抓住了主要矛盾：

如果能复用 prefix cache，那么整体成本和延迟会下降得更明显。

这是一种很成熟的性能观，不是盯着表面开销，而是盯着主成本中心。

### 2. 小的不稳定也会变成系统级损耗

文章里提到过一个很典型的例子：MCP 工具枚举顺序不固定，导致缓存未命中。

这个例子很精彩，因为它说明：

- 在 agent runtime 里，“顺序稳定性”不是小事
- 工具注册这种看似外围的实现细节，会直接影响推理成本
- 一旦会话很长，缓存 miss 的代价会被放大

这也是为什么成熟系统特别重视 deterministic behavior。

### 3. 配置更新尽量 append，而不是改旧内容

如果审批模式、sandbox 配置、cwd 改了，Codex 倾向于追加一条新的消息来表达变化，而不是回头修改旧消息。

这样做的目的不只是可追踪，更重要的是尽量保住前缀稳定性。

这是一种很聪明的折中：

- 允许状态变化
- 但减少对已有前缀的破坏
- 让缓存命中和状态演进同时成立

## 为什么这件事很“硬核”

因为这已经不是 prompt engineering 了，而是推理基础设施设计。

你在这里优化的不是一句提示词，而是：

- 会话状态的组织方式
- 工具系统的稳定性
- 配置变更的表达方式
- 请求构造策略

换句话说，Codex 是在围绕“缓存友好性”重塑 runtime 的行为。

## 如果你自己做 agent，可以直接借鉴什么

- 固定系统提示、工具定义和静态约束的顺序。
- 把易变内容尽量放到尾部。
- 避免中途重写历史消息。
- 对工具列表、配置项做稳定排序。
- 在设计新功能时，先问一句：这会不会破坏已有 prompt 前缀？

## 这个点的本质

它不是“利用缓存”这么简单，而是：

把缓存命中率提升为一等约束，再反过来塑造 runtime 的状态组织方式。

很多 agent 框架只有 loop，没有这一层意识；而这一层意识恰恰决定了系统在真实生产环境里能不能跑得久、跑得稳、跑得便宜。
