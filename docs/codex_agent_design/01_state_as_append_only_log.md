# 01. 把会话状态设计成 Append-Only Log

## 为什么这是一个真正的技术点

很多人实现 agent 时，会把“上下文”理解成一段不断增长的 prompt 字符串。这样做在 demo 里没问题，但系统一复杂就会遇到几个麻烦：

- 工具调用结果、模型思考、用户消息、系统约束混在一起，语义边界不清楚。
- UI 展示和 agent 内部状态不是同一个抽象，需要额外转换。
- 后续做调试、压缩、重放、断点恢复时，会很难定位“到底哪一步发生了什么”。

Codex 的一个重要设计是：不把状态看成一段字符串，而是看成一组有类型的 item。用户消息、assistant 消息、tool call、tool output、reasoning summary、instructions，都会以结构化项进入上下文。

这背后的价值在于，它把 agent 从“prompt 拼接器”提升成了“状态机”。

## 这个设计到底好在哪

### 1. 同一份状态可以服务多个目标

如果状态是结构化的，那么同一份数据就可以被不同模块使用：

- 给模型时，可以序列化成 Responses API 需要的 `input`。
- 给 UI 时，可以渲染成对话、工具执行记录、推理摘要。
- 给调试系统时，可以还原出每轮到底做了什么。
- 给压缩模块时，可以挑出高价值项进行摘要或合并。

这说明它不是单纯为了“喂给模型”，而是为了支撑整个 agent runtime。

### 2. 更接近 event sourcing 的思路

如果把每次用户输入、模型输出、工具调用和工具结果都当作事件追加到日志里，那么一次任务执行过程其实就是一条事件流。

这个思路有几个明显好处：

- 可追踪：你可以回看完整执行轨迹。
- 可重放：理论上可以从历史状态重新构建某个时刻的上下文。
- 可审计：在企业环境里，能解释 agent 做过什么很重要。
- 可恢复：中断后更容易从中间状态恢复。

对 agent 来说，这比“最后拼出一个 prompt 发出去”要高级得多。

### 3. 工具调用不再是 prompt 外挂，而是状态的一部分

很多简单 agent 框架会把工具调用看成外部逻辑：

1. 模型输出一段函数调用
2. 框架执行函数
3. 把结果塞回 prompt

Codex 的设计更统一。工具请求和工具结果本身就是会话状态的一部分。这样做的意义是：

- 工具调用链条天然可见
- 后续轮次可以直接消费前一轮工具结果
- 工具层和对话层不是松散拼接，而是共享一份状态模型

这会让系统更容易演进，比如后续加入压缩、缓存、分支执行、审计都更自然。

## 为什么这不是一个“显而易见”的设计

因为很多人会直觉上把 agent 理解成：

用户输入 -> 模型输出 -> 调工具 -> 把结果再发给模型

这个流程没错，但它天然偏“控制流”视角。

Codex 真正更成熟的地方在于，它采用的是“状态视角”：

不是“我下一步该做什么”，而是“当前系统状态由哪些结构化事实组成，这些事实如何被持续追加和消费”。

这是从脚本式 agent 到 runtime 式 agent 的关键一步。

## 如果你自己实现 agent，这个点最值得抄什么

- 不要只维护一段最终 prompt，维护结构化消息列表。
- 给每条状态附加类型，让用户消息、工具调用、工具结果、系统约束彼此可区分。
- 保持状态追加式演进，少做原地改写。
- 从一开始就考虑“UI 怎么看这份状态”“压缩器怎么读这份状态”“调试器怎么重放这份状态”。

## 一句话总结

这个设计的精华不在“把消息存下来”，而在于：

它把 agent 运行过程建模成了结构化、可追加、可重放的状态日志，这使得长链路 agent 具备了真正的工程可维护性。
